# ADR-002: ZFS Storage Strategy for HLH-Docker

**Status:** Adopted
**Date:** 2026-06-08
**Author:** Kevin
**System:** HLH-Docker LXC (vmid 102)
**Platform:** Proxmox 9.2, ZFS pool `RaidZ1-6TB`

## Context

The Docker LXC needs persistent storage that survives LXC destroy/recreate cycles (nuke-and-deploy
workflow). Docker images, layers, named volumes, and Dockhand metadata must not be lost when the
LXC container is recreated. The Proxmox host uses a ZFS pool (`RaidZ1-6TB`) for all LXC storage.

## Decision

Use **ZFS bind-mounts** configured in the OpenTofu LXC resource via `mount_block` blocks. Pre-existing
ZFS datasets on the Proxmox host are bind-mounted into the LXC at specific mount points. The datasets
must exist before `tofu apply` — OpenTofu does not manage dataset creation, only the bind-mount
configuration.

### Bind Mount Layout

| ZFS Dataset | LXC Mount Point | Purpose |
|---|---|---|
| `RaidZ1-6TB/hlh-docker/docker-data` | `/var/lib/docker` | Docker engine data (images, layers, volumes, build cache) |
| `RaidZ1-6TB/hlh-docker/dockhand-data` | `/srv/dockhand/data` | Dockhand container metadata and configuration |

### Implementation

OpenTofu `mount_block` blocks in `opentofu/main.tf`:

```hcl
mount_block {
  datastore_id = "RaidZ1-6TB"
  source       = "RaidZ1-6TB/hlh-docker/docker-data"
  mp           = "/var/lib/docker"
  content      = "volume"
}

mount_block {
  datastore_id = "RaidZ1-6TB"
  source       = "RaidZ1-6TB/hlh-docker/dockhand-data"
  mp           = "/srv/dockhand/data"
  content      = "volume"
}
```

### Dataset Prerequisites

Before running `tofu apply`, the following datasets must exist on the Proxmox host:

```bash
zfs create RaidZ1-6TB/hlh-docker/docker-data
zfs create RaidZ1-6TB/hlh-docker/dockhand-data
```

These are already created. If they are missing, create them manually on the Proxmox host before deploy.

### Dataset Lifecycle

- Datasets persist on the Proxmox host regardless of LXC lifecycle (created/destroyed/started/stopped).
- Destroying the LXC via OpenTofu **does not** destroy the ZFS datasets.
- Recreating the LXC via OpenTofu will re-attach the existing datasets automatically.
- Docker and Dockhand data survives across nuke-and-deploy cycles.

### Naming Convention

```
RaidZ1-6TB/<app-name>/<purpose>
```

Examples:
- `RaidZ1-6TB/hlh-docker/docker-data` — Docker engine data
- `RaidZ1-6TB/hlh-docker/dockhand-data` — Dockhand metadata

### Reserved Datasets (Not Currently Mounted)

| Dataset | Status |
|---|---|
| `RaidZ1-6TB/hlh-docker/rootfs` | Reserved, not mounted |

## Rationale

- **Bind-mounts** are simpler than ZFS subvolume passthrough for unprivileged LXCs.
- Datasets are independent of the LXC — they survive `pct destroy` and `tofu destroy`.
- No nested ZFS or complex mount propagation needed.
- Unprivileged LXC supports bind-mounts with `content: "volume"` and `mp` set.

## Consequences

### Positive
- Nuke-and-deploy workflow preserves all Docker/Dockhand state.
- Datasets are independently manageable (snapshots, clones, migration).
- Simple, transparent mount configuration.

### Negative
- Datasets must be created manually before `tofu apply` (not yet managed by OpenTofu).
- If a dataset is accidentally destroyed on the host, all Docker/Dockhand data is lost.
- `content: "volume"` semantics mean Proxmox treats the source as a ZFS volume (not a directory).

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| ZFS subvolume passthrough (`content: "volume"` with `mp` as new subvol) | OpenTofu manages subvolume lifecycle — destroyed with LXC |
| Docker storage driver overlay2 on rootfs | Data lost on LXC destroy |
| Host-level bind mounts via `/etc/pve/storage.cfg` | More complex, requires storage definition |
| LVM-thin on Proxmox | Adds another layer of indirection over ZFS |
